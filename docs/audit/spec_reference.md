## §1 — Session Definitions

### S1.1 Session Type Enum

| Session | Numeric ID | Open Time (ET) | Timezone Reference |
|---------|-----------|----------------|-------------------|
| NY | 1 | 09:30 | America/New_York |
| LON | 2 | 08:00 | Europe/London (displayed in ET internally) |
| APAC | 3 | [SPEC AMBIGUOUS: exact APAC open time not specified in Online spec] | [converted to ET internally] |

**System timezone rule:** All internal timestamps stored in `America/New_York`. System clock assertion enforced at session start (V3 line 143):
```
system_tz = "America/New_York"
ASSERT system_clock_timezone() == system_tz
```

**Max simultaneous sessions:** 3 (NY, LON, APAC) -- fixed constant (V3 line 1597).

### S1.2 session_match() Logic

```
FUNCTION session_match(asset, session_id):
    session_key = {1: "NY", 2: "LON", 3: "APAC"}[session_id]
    RETURN P3-D00[asset].session_hours.get(session_key) is not None
```

Source: V3 lines 534-537. Asset-to-session mapping is stored in `P3-D00[asset].session_hours` -- a dict keyed by session name. If the asset has an entry for that session, it participates.

### S1.3 Per-Session Asset Mapping

### S1.4 Opening Range (OR) Parameters

OR formation window is per-model, not global:

```
FUNCTION get_or_window_minutes(model_id):
    model_def = D-09[model_id]
    RETURN model_def.strategy_params.get("OR_window_minutes", 15)  # default 15 min
```

Source: V3 lines 596-600. The `m` parameter from the locked strategy (P2-D06) identifies the model, and `D-09[m].strategy_params.OR_window_minutes` gives the asset-specific OR duration.

AIM-15 (Opening Volume) uses this window:
```
or_minutes = get_or_window_minutes(locked_strategies[u].m)
features[u].opening_volume_ratio = volume_first_N_min(u, or_minutes) / avg_volume_first_N_min(u, or_minutes)
```

## §2 — Online Blocks 1–9

### Block 1: Pre-Session Data Ingestion

- **PG ID:** P3-PG-21 (`data_ingestion_A`)
- **Purpose:** Load all required data and compute pre-session features for the upcoming evaluation. Session-aware: different assets evaluated at different session opens.
- **Trigger:** Session open (orchestrator detects `session_opening_now(ss)`)
- **Shared vs Per-User:** SHARED -- computed once per session, same result for all users.

**Inputs:**
| Source | Key Fields |
|--------|-----------|
| P3-D00 (asset_universe) | `captain_status`, `session_hours`, `data_quality_flag`, `roll_calendar`, `data_sources`, `point_value`, `p1_data_path`, `p2_data_path` |
| P3-D01 (aim_model_states) | `aim_id`, `status` (ACTIVE/BOOTSTRAPPED/ELIGIBLE) |
| P3-D02 (aim_meta_weights) | `aim_id`, `inclusion_flag`, `inclusion_probability` |
| P3-D05 (ewma_states) | `asset`, `regime`, `win_rate`, `avg_win`, `avg_loss`, `latest_sigma` |
| P3-D08 (tsm_state) | Account TSM configurations |
| P3-D12 (kelly_params) | `asset`, `regime`, `session`, `kelly_full`, `shrinkage_factor`, `sizing_override` |
| P2-D06 (locked_strategies) | `asset`, `m`, `k`, threshold, strategy params |
| P2-D07 (regime_models) | Trained classifiers per asset, `model_type`, `feature_list` |
| External data feeds | VIX, VXV, options chains, COT reports, economic calendar, live prices |

**Outputs:**
| Target | Key Fields |
|--------|-----------|
| `active_assets` list | Filtered by `captain_status == "ACTIVE"` AND `session_match()` |
| `features[asset]` dict | Per-AIM computed features (see feature list below) |
| All loaded state objects | Passed through to Blocks 2-6 |
| P3-D17 (system_params) | `data_quality_log` entry appended |
| P3-D00 (asset_universe) | `data_quality_flag` updated, `captain_status` may change to DATA_HOLD/ROLL_PENDING |

**Key Logic:**

1. **Asset filtering:** `captain_status == "ACTIVE"` AND `session_match(asset, session_id)`. Empty active set -> RETURN early.

2. **Data Moderator (Step 1b)** -- four pre-ingestion sanity checks:
   - **Price bounds:** >5% deviation from prior close -> `DATA_HOLD` status, P2_HIGH incident, asset excluded.
   - **Volume sanity:** Zero volume -> `ZERO_VOLUME` flag. >10x average -> `VOLUME_EXTREME`. Otherwise `CLEAN`.
   - **Missing data:** Each AIM's required features checked against data source availability. Missing -> `STALE_FEATURE` flag. Uses last known value as fallback.
   - **Timestamp validation:** Missing timezone offset -> P2_HIGH incident, asset skipped.

3. **Contract roll calendar (Step 1c):**
   - Roll date reached + not confirmed -> `ROLL_PENDING` status, CRITICAL notification, signals paused.
   - Roll within 3 days -> HIGH notification warning.
   - `ROLL_PENDING` assets excluded from `active_assets`.

4. **Feature computation (Step 4):** 13 AIM feature groups computed per asset:
   - AIM-01: VRP (`compute_vrp`, `compute_overnight_vrp`) -- RV minus IV
   - AIM-02: Skew (`compute_put_call_ratio`, `compute_dotm_otm_put_spread`) -- PCR and 10d-25d put spread
   - AIM-03: GEX (`compute_dealer_net_gamma`) -- dealer net gamma from options OI
   - AIM-04: IVTS -- `vix_close_yesterday / vxv_close_yesterday` (critical regime filter)
   - AIM-06: Economic calendar (`check_economic_calendar`, `min_distance_to_event`) -- event tiers 1-4
   - AIM-07: COT positioning (`latest_smi_polarity`, `speculator_z_score`) -- institutional vs retail
   - AIM-08: Cross-asset correlation (`rolling_20d_correlation("ES", "CL")`, z-scored vs 252d)
   - AIM-09: Cross-asset momentum (`compute_cross_asset_momentum`, MACD-based, lookback=21)
   - AIM-10: Calendar effects (`is_within_opex_window` +/- 2 trading days of 3rd Friday, day_of_week)
   - AIM-11: Regime warning (VIX z-score vs 252d, VIX daily change z-score vs 60d; CL basis for CL)
   - AIM-12: Dynamic costs (`get_live_spread`, spread z-score vs 60d)
   - AIM-15: Opening volume ratio (first N min volume / 20-day avg first N min volume)
   - Overnight return: `(open_today / close_yesterday) - 1`

**Error Handling:** Data feed unavailable -> use last known values, flag staleness in signal, alert.

### Block 2: Regime Probability Computation

- **PG ID:** P3-PG-22 (`regime_probability_A`)
- **Purpose:** Predict LOW_VOL/HIGH_VOL regime probabilities for each asset using P2-trained classifiers.
- **Trigger:** Immediately after Block 1 completes.
- **Shared vs Per-User:** SHARED -- computed once per session.

**Inputs:**
| Source | Key Fields |
|--------|-----------|
| Block 1 output | `features`, `active_assets` |
| P2-D07 (regime_models) | Per-asset classifier (`model_type`, `feature_list`, `pettersson_threshold`) |
| P3-D05 (ewma_states) | `latest_sigma` (for BINARY_ONLY fallback) |

**Outputs:**
| Target | Key Fields |
|--------|-----------|
| `regime_probs[asset]` | `{LOW_VOL: float, HIGH_VOL: float}` |
| `regime_uncertain[asset]` | `True` if `max(probs) < 0.6` |
| Redis cache | `regime_probs` cached |

**Key Logic:**

Two classifier paths based on P2 complexity tier:

1. **BINARY_ONLY (C4 assets):** Pettersson binary threshold rule.
   ```
   sigma_today = compute_realised_vol(features[u])
   IF sigma_today > classifier.pettersson_threshold:
       regime_probs = {HIGH_VOL: 1.0, LOW_VOL: 0.0}
   ELSE:
       regime_probs = {HIGH_VOL: 0.0, LOW_VOL: 1.0}
   ```
   [SOURCE: Obsidian doc 33 uses label `BINARY_THRESHOLD` for same concept]

2. **C1/C2/C3 assets:** Trained classifier (XGBoost / LogReg per P2 tier).
   ```
   feature_vector = extract_classifier_features(u, features)
   regime_probs[u] = classifier.predict_proba(feature_vector)
   ```
   Feature alignment enforced via `P2-D07.feature_list`.

3. **Uncertainty flag:** `regime_uncertain[u] = (max_prob < 0.6)` -- triggers robust Kelly fallback in Block 4 L3.

**Error Handling:** Classifier fails -> use 50/50 regime probs, flag uncertainty, force robust Kelly.

### Block 3: AIM Aggregation (MoE/DMA)

- **PG ID:** P3-PG-23 (`aim_aggregation_A`)
- **Purpose:** Compute weighted combined modifier from all active AIMs using Dynamic Model Averaging (DMA) inclusion probabilities.
- **Trigger:** Immediately after Block 2 completes.
- **Shared vs Per-User:** SHARED -- computed once per session.

**Inputs:**
| Source | Key Fields |
|--------|-----------|
| Block 1 output | `features`, `aim_states` |
| P3-D01 (aim_model_states) | `aim_id`, `status` (must be ACTIVE) |
| P3-D02 (aim_meta_weights) | `aim_id`, `inclusion_flag`, `inclusion_probability` (DMA weights from Offline) |

**Outputs:**
| Target | Key Fields |
|--------|-----------|
| `combined_modifier[asset]` | Clamped to `[0.5, 1.5]`, default 1.0 if no active AIMs |
| `aim_breakdown[asset]` | Per-AIM: `{modifier, confidence, reason_tag, dma_weight}` |
| Redis cache | `combined_modifier`, `aim_breakdown` cached |

**Key Logic:**

1. **Per-AIM computation:** For each AIM (1-15), if `status == ACTIVE` AND `inclusion_flag == True`:
   - Call `compute_aim_modifier(aim_id, features, asset)` (logic per AIM in AIMRegistry.md Part J)
   - Per-AIM modifier clamped to `[0.5, 1.5]` (FLOOR/CEILING)

2. **MoE weighted aggregation:**
   ```
   total_weight = SUM(dma_weight for active AIMs)
   combined_modifier = SUM(modifier * dma_weight / total_weight)
   combined_modifier = clamp(combined_modifier, 0.5, 1.5)
   ```

3. **AIM-16 HMM session budget weights** [SOURCE: Obsidian doc 33]:
   ```
   IF aim_states[16].status == ACTIVE:
       session_budget_weights = hmm_inference(P3-D26, features, session_id)
   ELSE:
       session_budget_weights = {NY: 1.0, LON: 1.0, APAC: 1.0}  # equal
   ```
   [SPEC AMBIGUOUS: AIM-16 HMM session budget weights appear in Obsidian doc 33 Block 3 but are absent from V3 Block 3. V3's Block 5 does reference `session_budget_weights` indirectly via the Obsidian doc's Block 5 daily_budget concept. The V3 spec iterates AIMs 1-15, not 1-16.]

**Error Handling:** All AIMs fail -> `combined_modifier = 1.0`, proceed with base Kelly.

### Block 4: Kelly Sizing Under Constraints

- **PG ID:** P3-PG-24 (`kelly_sizing_A`)
- **Purpose:** Compute optimal contract sizing per asset per account for the current user. 7-layer pipeline (L1-L7) plus silo drawdown pre-check.
- **Trigger:** After shared Blocks 1-3 complete, runs per-user.
- **Shared vs Per-User:** PER-USER -- uses this user's capital silo and account configs.

**Inputs:**
| Source | Key Fields |
|--------|-----------|
| Block 2 output | `regime_probs`, `regime_uncertain` (shared) |
| Block 3 output | `combined_modifier` (shared) |
| P3-D12 (kelly_params) | `asset`, `regime`, `kelly_full`, `shrinkage_factor` (shared) |
| P3-D05 (ewma_states) | `asset`, `regime`, `win_rate`, `avg_win`, `avg_loss` (shared) |
| P3-D08 (tsm_state) | `classification.risk_goal`, `classification.category`, `max_drawdown_limit`, `current_drawdown`, `max_daily_loss`, `daily_loss_used`, `max_contracts`, `scaling_plan`, `pass_probability`, `instrument_permissions`, `current_balance`, `overnight_allowed`, `commission_per_contract` |
| P3-D16 (capital_silos) | `user_id`, `total_capital`, `starting_capital`, `accounts[]`, `risk_allocation.user_kelly_ceiling`, `risk_allocation.max_portfolio_risk_pct` |
| P3-D17 (system_params) | `max_silo_drawdown_pct` (default 0.30), `tsm_budget_divisor_default` (default 20) |
| P2-D06 (locked_strategies) | `threshold` (SL distance in points) |
| P3-D00 (asset_universe) | `point_value` |

**Outputs:**
| Target | Key Fields |
|--------|-----------|
| `final_contracts[asset][account_id]` | Integer contract count (>= 0) |
| `account_recommendation[asset][account_id]` | `TRADE`, `SKIP`, `BLOCKED`, or `REDUCED_TO_ZERO` |
| `account_skip_reason[asset][account_id]` | Human-readable reason string or None |

**Key Logic -- 7-Layer Pipeline:**

0. **Silo drawdown pre-check:** If `(1 - total_capital/starting_capital) > max_silo_drawdown_pct` (default 30%), ALL trading halted for this user. CRITICAL notification.

1. **L2 -- Blended Kelly:** `f = SUM(regime_prob[r] * kelly_full[asset][r])` across LOW_VOL, HIGH_VOL.

2. **L3 -- Shrinkage:** `f *= shrinkage_factor` (from Offline Block 8, Paper 217).

3. **L4 -- Robust fallback:** If `regime_uncertain[u]`:
   ```
   robust_kelly = compute_robust_kelly(return_bounds, "moment_constraints")
   f = min(f, robust_kelly)
   ```
   Robust Kelly uses distributional min-max (Paper 218), capped at half-Kelly.

4. **L5 -- AIM modifier:** `f *= combined_modifier[u]`

5. **L5b -- User Kelly ceiling:** `f = min(f, user_silo.risk_allocation.user_kelly_ceiling)`

6. **L6 -- Account adjustment by risk_goal:**
   | risk_goal | Multiplier |
   |-----------|-----------|
   | `PASS_EVAL` (pass_prob < 0.5) | 0.5 |
   | `PASS_EVAL` (pass_prob < 0.7) | 0.7 |
   | `PASS_EVAL` (pass_prob >= 0.7) | 0.85 |
   | `PRESERVE_CAPITAL` | 0.5 |
   | `GROW_CAPITAL` | 1.0 (full) |

7. **L7 -- TSM hard constraints:** Two paths by account category:

   **PROP accounts** (`PROP_EVAL`, `PROP_FUNDED`, `PROP_SCALING`):
   ```
   daily_budget = remaining_mdd / max(days_remaining, 1)  # timed eval
               OR remaining_mdd / budget_divisor           # no deadline (default divisor=20)
   max_by_mdd = floor(daily_budget / (SL * point_value))
   max_by_mll = floor((max_daily_loss - daily_loss_used) / (SL * point_value))
   tsm_cap = min(max_by_mdd, max_by_mll, max_contracts)
   ```
   Plus scaling plan tier cap if active.

   **BROKER accounts** (`BROKER_RETAIL`, `BROKER_INSTITUTIONAL`):
   ```
   tsm_cap = floor(current_balance / (margin_per_contract * buffer))  # buffer default 1.5
   ```

8. **Final contracts:** `min(floor(kelly_fraction * capital / risk_per_contract), tsm_cap)`, floored at 0.

9. **Portfolio risk cap (Step 7):** Total risk across all accounts for one asset capped at `max_portfolio_risk_pct * total_capital`. Scale down proportionally if exceeded.

10. **Level 2 sizing override (Step 8):** If Offline decay detection set `sizing_overrides[asset]`, multiply all account contracts. May reduce to zero -> `REDUCED_TO_ZERO` recommendation.

**Error Handling:** Kelly computation error -> fall back to minimum position size (1 contract) or skip.

### Circuit Breaker (PG-27B) -- 5-Layer Screen [SOURCE: Obsidian doc 33]

[SOURCE: Obsidian doc 33. This block appears as a standalone screen in Obsidian's pseudocode but is NOT present as a separate block in the V3 Online spec. - **Purpose:** Multi-layer risk screen applied to allocated signals before output.
- **Trigger:** After Block 5 trade selection, before Block 6 signal output.

### Block 5: Universe-Level Trade Selection

- **PG ID:** P3-PG-25 (`trade_selection_A`)
- **Purpose:** Select which assets this user should trade this session, considering expected edge, correlation, and position limits.
- **Trigger:** After Block 4 (Kelly sizing) completes for this user.
- **Shared vs Per-User:** PER-USER -- selection decisions are user-specific (different contract sizes after Block 4).

**Inputs:**
| Source | Key Fields |
|--------|-----------|
| Block 4 output | `final_contracts`, `account_recommendation`, `account_skip_reason` (this user's) |
| P3-D05 (ewma_states) | `win_rate`, `avg_win`, `avg_loss` per regime (shared) |
| Block 2 output | `regime_probs` (shared) |
| P3-D07 (correlation_matrix) | Cross-asset correlations from AIM-08 (shared) |
| P3-D16 (capital_silos) | `risk_allocation.correlation_threshold` (default 0.7), `risk_allocation.max_simultaneous_positions` |

**Outputs:**
| Target | Key Fields |
|--------|-----------|
| `selected_trades` | Ordered list of assets with positive edge and contracts |
| `score[asset]` | `expected_edge * max_contracts_this_user` |
| `expected_edge[asset]` | `win_rate * avg_win - (1-win_rate) * avg_loss` ($/contract) |
| `final_contracts` (modified) | May be reduced by correlation adjustment or position limit |
| `account_recommendation` (reconciled) | Updated to `SKIP` if Block 5 zeroed contracts |

**Key Logic:**

1. **Expected edge:** Per-contract dollar expectation in dominant regime.
   ```
   expected_edge[u] = ewma[u][regime].win_rate * avg_win - (1 - win_rate) * avg_loss
   ```

2. **Score:** `expected_edge * max_contracts_this_user` (user-specific).

3. **Correlation adjustment:** For each pair with `corr > corr_threshold` (default 0.7), the lower-scored asset gets its contracts halved (floored).

4. **Position limit:** Assets ranked by score descending. Beyond `max_simultaneous_positions`, all contracts zeroed.

5. **Recommendation reconciliation:** If Block 5 zeroed a previously `TRADE` asset, update to `SKIP` with reason `"Removed by portfolio-level constraint"`.

6. **Final selection:** Assets with `max_contracts > 0` AND `expected_edge > 0`.

**Error Handling:** No assets pass -> generate "NO TRADE" signal, log reason.

### Block 5B: Signal Quality Gate

- **PG ID:** P3-PG-25B (`signal_quality_gate_A`)
- **Purpose:** Filter selected trades by minimum quality threshold. Below-threshold signals logged as `AVAILABLE_NOT_RECOMMENDED`. Filter only -- does NOT modify passing signals.
- **Trigger:** After Block 5 completes for this user.
- **Shared vs Per-User:** PER-USER (quality score uses shared edge + modifier, but trade_count may vary).

**Inputs:**
| Source | Key Fields |
|--------|-----------|
| Block 5 output | `selected_trades`, `expected_edge`, `score` |
| Block 3 output | `combined_modifier` (shared) |
| Block 2 output | `regime_probs` (shared) |
| P3-D05 (ewma_states) | (shared) |
| P3-D03 (trade_outcome_log) | `trade_count` for data maturity ramp |
| P3-D17 (system_params) | `quality_hard_floor`, `quality_ceiling` |

**Outputs:**
| Target | Key Fields |
|--------|-----------|
| `recommended_trades` | Assets passing quality gate |
| `available_not_recommended` | Assets below threshold (logged for transparency) |
| `quality_results[asset]` | `{quality_score, quality_multiplier, passes_gate, edge, modifier, data_maturity, trade_count}` |
| P3-D17 session_log | Session quality summary appended |

**Key Logic:**

1. **Quality score:** `quality_score = expected_edge * combined_modifier * data_maturity`
   - `data_maturity = min(1.0, trade_count / 50)` -- ramps 0->1 over first 50 trades.

2. **Confidence-graduated sizing (SPEC-A7):**
   ```
   IF quality_score < hard_floor:    quality_multiplier = 0.0, blocked
   ELSE:                             quality_multiplier = min(1.0, quality_score / quality_ceiling)
   ```

3. **Threshold parameters (OPEN -- require calibration):**
   | Parameter | Default | Range | Stored In |
   |-----------|---------|-------|-----------|
   | `quality_hard_floor` | 0.003 | [0.001, 0.05] | P3-D17.system_params |
   | `quality_ceiling` | 0.010 | [0.001, 0.05] | P3-D17.system_params |
   | `minimum_quality_threshold` | 0.005 | [0.001, 0.05] | P3-D17.system_params (alias for hard_floor) |

   **Calibration method:** Set threshold at 20th percentile of quality_scores among historically profitable P1/P2 trades (~80% of profitable signals pass).

   **FIXED after calibration** -- does NOT adapt dynamically (prevents overfitting/look-ahead bias).

**Error Handling:** Quality score computation fails -> pass all `selected_trades` through unfiltered (fail-open). Log warning. Alert ADMIN.

### Block 7: Intraday Position Monitoring

- **PG ID:** P3-PG-27 (`intraday_monitor_A`)
- **Purpose:** Monitor all open positions continuously. Track P&L, TP/SL proximity, market conditions. Positions are per-user; all notifications routed to position owner.
- **Trigger:** Continuous while any position is open across any user. Poll every 10 seconds.
- **Shared vs Per-User:** PER-USER -- each position belongs to a user's account.

**Inputs:**
| Source | Key Fields |
|--------|-----------|
| Open positions | `asset`, `entry_price`, `direction`, `contracts`, `tp_level`, `sl_level`, `user_id`, `account`, `risk_amount`, `aim_breakdown`, `combined_modifier`, `regime_state`, `session` |
| Live market data | Current price, VIX |
| P3-D00 (asset_universe) | `point_value` |
| P3-D08 (tsm_state) | `overnight_allowed`, `trading_hours`, `commission_per_contract` |

**Outputs:**
| Target | Key Fields |
|--------|-----------|
| P3-D03 (trade_outcome_log) | Full trade outcome record with `user_id`, `gross_pnl`, `commission`, `pnl` (NET), `slippage`, `outcome`, `regime_at_entry`, `aim_breakdown_at_entry` |
| P3-D16 (capital_silos) | Capital silo updated with NET PnL |
| Redis `captain:trade_outcomes` | Trade outcome event -> triggers Offline learning |
| GUI position updates | Per-user (each user sees only their own positions) |
| Notifications | Routed to `pos.user_id` |

**Key Logic:**

1. **P&L tracking:**
   ```
   current_pnl = (price - entry) * direction * contracts * point_value
   pnl_pct = current_pnl / risk_amount
   ```

2. **TP/SL proximity alerts:**
   - TP within 10% of distance -> HIGH notification to position owner.
   - SL within 10% of distance -> CRITICAL notification to position owner.

3. **Market condition alerts:**
   - VIX z-score > 2.0 (vs 60d trailing) -> HIGH alert.
   - Regime shift detected mid-trade -> CRITICAL alert.

4. **Position resolution triggers:**
   - TP hit, SL hit, TIME_EXIT (no overnight), MANUAL_CLOSE.

5. **resolve_position():** Records trade outcome with:
   - **Commission chain:** API fill -> TSM config -> user notification (fallback to 0 + notify).
   - **Entry price chain:** API fill -> manual GUI input -> notification (signal price as interim).
   - **PnL is NET** (after commissions) -- this feeds the Offline learning loop.
   - **Slippage** computed: `(actual_entry - signal_entry) * direction * contracts * point_value`.

6. **TIME EXIT:** For accounts with `overnight_allowed == false`, forced close 5 minutes before `trading_hours` end. CRITICAL notification.

**Error Handling:**
- Price feed drops -> alert immediately, use last known price, flag to user.
- `resolve_commission` returns 0 -> trade continues, user notified, PnL temporarily gross.
- `resolve_actual_entry_price` returns None -> signal price used as interim, slippage=None, AIM-12 skips this trade.

- **PG ID:** P3-PG-28 (`network_concentration_monitor_A`)
- **Purpose:** Aggregate exposure across all users to detect network-level concentration. Advisory/alerting only -- does NOT modify signals. Alerts sent to ADMIN users.
- **Trigger:** Runs ONCE per session AFTER all per-user deployment loops complete. Only runs if `len(active_users) > 1`.
- **Shared vs Per-User:** NETWORK-LEVEL (aggregates across all users).

**Inputs:**
| Source | Key Fields |
|--------|-----------|

**Key Logic:**

### S3.5 AIM Aggregation Method

**Dynamic Model Averaging (DMA)** is the primary aggregation framework:
- Each AIM is a "predictor" in the DMA system.
- DMA dynamically updates inclusion probabilities via a **forgetting factor** (default 0.99).
- Output: per-AIM dynamic inclusion probability -> automatically weights/suppresses AIMs.
- Combined modifier = weighted average of AIM outputs using DMA inclusion probabilities.
- HDWM architecture ensures seed diversity persists across regime changes.

```
FOR EACH trading day t:
    aim_outputs = [AIM_1.modifier, ..., AIM_15.modifier]
    model_probs = DMA_update(aim_outputs, forgetting_factor=0.99, prior_probs=yesterday_probs)
    combined_modifier = weighted_average(aim_outputs, weights=model_probs)
    final_size = kelly_fraction * combined_modifier
```

Source: AIM_Extractions.md System 1 Design Conclusions + Obsidian doc 31 section 4.

### AIM-01: Volatility Risk Premium Monitor

- **Tier:** 2
- **Seed Type:** Options
- **Paper/Source:** Kang & Pan (2015), Papagelis & Dotsis (2025), Andersen/Fusari/Todorov (2017, 2020), Johannes/Kaeck/Seeger (2024)

- **Warm-Up:** 120 trading days for z-score baseline (HAR regression needs 250d of RV, but VIX-based proxy available from day 1).

**Thresholds (Obsidian doc 31):**
| Condition | Modifier | Reason Tag |
|-----------|----------|------------|
| `vrp_z > 1.5` | 1.15 | VRP_HIGH_POSITIVE |

| `vrp_z < -1.0` | 0.85 | VRP_NEGATIVE |
| else | 1.0 | VRP_NEUTRAL |
| Overnight VRP elevated AND base >= 1.0 | base + 0.05 (max 1.5) | +OVERNIGHT_ELEVATED |

**Key Formula:**
```
VRP = E[RV] - IV   (positive = IV cheap relative to expected realized)
vrp_z = z_score(vrp, trailing_120d_vrp[asset])
confidence = min(abs(vrp_z) / 2.0, 1.0)
```

**AIM_Extractions refinement:** Overnight VRP is the operationally relevant signal (Captain runs pre-open). Monday mornings carry extra weekend VRP uncertainty -> additional 0.95 caution factor. Jump tail component from deep OTM puts is an advanced supplement.

**PG-02: DMA Meta-Learning Update** (Paper 187) -- Called after each trade outcome:

```
P(AIM_a | data) ∝ P(AIM_a | data_{t-1})^λ × L(x_t | AIM_a)
```

- **Forgetting factor λ:** 0.99 (OPEN PARAMETER, range 0.95-0.99). Higher = slower adaptation. λ=0.99 ~ 100 trades effective memory.
- **Magnitude-weighted likelihood (SPEC-A9):** z-score normalised, range [0, 1]:
  - AIM said "size up" (modifier > 1.0): win -> high L (0.5 + 0.5×z/3); loss -> low L (0.5 - 0.5×z/3)
  - AIM said "size down" (modifier < 1.0): loss -> high L; win -> low L
  - Neutral (modifier == 1.0): L = 0.5
  - z = min(|pnl_pc| / max(avg_win_or_loss, 0.01), 3.0) -- capped at 3 to prevent outlier domination
- **Normalisation:** Mandatory after every update. Inclusion probabilities must sum to 1.0 across all active AIMs.
- **Inclusion threshold:** 0.02 (P3-D17.system_params). Below this, inclusion_flag = False.
- **Critical:** Must use modifier from TRADE TIME, not current modifier. P3-D03 stores aim_breakdown_at_entry for this purpose.

### Block 2: Strategy Decay Detection

- **PG IDs:** P3-PG-05 (bocpd_decay_monitor_A), P3-PG-06 (cusum_decay_monitor_A), P3-PG-07 (cusum_bootstrap_calibrate_A), P3-PG-08 (decay_response_handler_A)
- **Purpose:** Monitors live trade outcomes for signs that the locked strategy's edge has deteriorated. Dual detection: BOCPD (primary) + distribution-free CUSUM (complementary).
- **Trigger:** PG-05 and PG-06 per trade outcome; PG-07 once at init then quarterly
- **Inputs:** P3-D03 (trade outcomes) | P3-D04 (prior BOCPD/CUSUM states) | P3-D05 (EWMA states)
- **Outputs:** P3-D04 (updated detector states, decay event log) | P3-D12 (sizing overrides on Level 2)
- **Shared vs Per-User:** SHARED INTELLIGENCE -- monitors strategy's market-level performance, not individual user capital

**PG-05: BOCPD** (Adams & MacKay 2007, Paper 231):
- Maintains posterior over run length r_t (time since last changepoint)
- Conjugate: Normal-Inverse-Gamma -> Student-t predictive
- Recursive update per asset: growth_prob = (1 - hazard_rate) × predictive; changepoint_prob = hazard_rate × predictive
- Changepoint probability cp = posterior[0] (mass at r=0)
- hazard_rate: 1/200 (range 1/100 to 1/500); max_run_length: 500
- **Level 2 trigger:** cp > 0.8
- **Level 3 trigger:** cp > 0.9 sustained for 5 consecutive days

**PG-06: CUSUM** (Paper 232) -- Two-sided, distribution-free:
- C_up = max(0, C_up_prev + pnl - k); C_down = max(0, C_down_prev - pnl - k)
- k = allowance (typically delta/2); h = sequential control limit (sprint-length dependent)
- Sprint length T_n: resets to 0 when both C_up and C_down are 0; increments otherwise
- **Level 2 trigger:** C_up > h_sequential OR C_down > h_sequential (then reset to 0)

**PG-07: Bootstrap Calibration** -- Init + quarterly:
- B=2000 bootstrap resamples from in-control trades
- Computes sequential control limits per sprint length j: h(j) = quantile(bootstrap_dist, 1 - 1/ARL_0)
- ARL_0 = 200 (average run length before false alarm)

**PG-08: Decay Response Handler:**

| Level | Condition | Action |
|-------|-----------|--------|
| **Level 2** | cp > 0.8 OR CUSUM breach | Autonomous sizing reduction: factor = max(0.5, 1.0 - (severity - 0.8) × 2.5). Writes to P3-D12.sizing_override[asset]. Notify GUI (AMBER) + Telegram (HIGH). |
| **Level 3** | cp > 0.9 sustained 5 days | Halt signals: P3-D00[asset].captain_status = "DECAYED". Notify GUI (RED) + Telegram (CRITICAL). Schedule P1/P2 rerun + AIM-14 auto-expansion. |

**Error Handling:** BOCPD numerical overflow -> reset run-length distribution, alert, use CUSUM as sole detector until resolved.

**PG-15: Kelly Parameter Update:**
1. Normalise to per-contract PnL: pnl_per_contract = pnl / contracts (removes sizing bias)
2. **SPEC-A12: Adaptive EWMA decay** -- alpha scales with BOCPD changepoint probability:

| cp_prob | effective_span | alpha | Regime |
|---------|---------------|-------|--------|
| < 0.2 | 30 | 0.065 | Stable -- slower learning |
| < 0.5 | 20 | 0.095 | Default |
| < 0.8 | 12 | 0.154 | Elevated instability |

   - win_rate = (1 - alpha) × win_rate + alpha × win
   - avg_win = (1 - alpha) × avg_win + alpha × win_size (on win)
   - avg_loss = (1 - alpha) × avg_loss + alpha × loss_size (on loss)
   - All values in per-contract dollar terms

4. Recompute Kelly fraction per [regime][session]:
   - b = W / L (dimensionless win/loss ratio)
   - **kelly_full = max(0, p - (1-p)/b)**

5. Shrinkage (Paper 217): shrinkage = max(0.3, 1.0 - estimation_variance). Floor at 30% Kelly.

## §5 — Kelly 7-Layer Sizing Pipeline

The Kelly pipeline spans Offline (parameter computation, Layers 1+3) and Online (application, Layers 2+4-7). Offline Block 8 produces the raw regime-conditional Kelly fractions; Online Block 4 blends and constrains them into a final contract count.

**Data flow:** P3-D03 -> P3-D05 -> P3-D12 -> Online Block 4 -> contract count

### Layer 1: Regime-Conditional Raw Kelly

- **Purpose:** Compute per-contract Kelly fraction for each [asset][regime][session] cell
- **Where:** Offline Block 8 (PG-15)
- **Inputs:** P3-D03 (trade outcomes), P3-D05 (EWMA states), P3-D04 (BOCPD cp probability)
- **Formula:**
  ```
  b = W / L                         (win/loss ratio, dimensionless)
  kelly_full = max(0, p - (1-p)/b)  (Kelly criterion)
  ```
  Where p = EWMA win rate, W = EWMA avg win (per-contract $), L = EWMA avg loss (per-contract $)
- **EWMA update:** Session-specific, regime-conditional. Adaptive alpha driven by BOCPD changepoint probability (SPEC-A12):
  - cp < 0.2 -> span=30 (stable); cp < 0.5 -> span=20; cp < 0.8 -> span=12; cp >= 0.8 -> span=8
  - alpha = 2 / (effective_span + 1)
- **Output:** P3-D12[asset][regime][session].kelly_full
- **Dataset Refs:** P3-D03, P3-D04, P3-D05, P3-D12
- **Constraints:** kelly_full floored at 0 (no negative sizing)

### Layer 2: Regime Blend

- **Purpose:** Blend regime-conditional Kelly fractions using regime probability weights
- **Where:** Online Block 4 (Paper 219, MacLean & Zhao)
- **Inputs:** P3-D12 (per-regime Kelly fractions), P2-D07 regime probabilities, current session
- **Formula:**
  ```
  f_blended = Σ P(regime_r) × f*_r
  ```
  Where P(regime_r) comes from Online Block 2 regime probability output, f*_r = kelly_full for that regime
- **Output:** Single blended Kelly fraction per asset for current session
- **Dataset Refs:** P3-D12, P2-D07 (via Online Block 2)
- **Constraints:** None additional (inherits L1 floor)

### Layer 3: Shrinkage

- **Purpose:** Reduce Kelly fraction proportional to estimation uncertainty. Prevents oversizing with limited data.
- **Where:** Offline Block 8 (PG-15) computes shrinkage factor; Online Block 4 applies it
- **Inputs:** P3-D05 (EWMA states), P3-D03 (trade count)
- **Formula:**
  ```
  shrinkage = max(0.3, 1.0 - estimation_variance)
  f_shrunk = f_blended × shrinkage
  ```
  (Paper 217)
- **Output:** P3-D12[asset].shrinkage_factor
- **Dataset Refs:** P3-D05, P3-D12
- **Constraints:** Shrinkage floored at 0.3 (never less than 30% of raw Kelly). Approaches 1.0 as data accumulates.

### Layer 4: Robust Fallback

- **Purpose:** Additional protection when regime classification is uncertain
- **Where:** Online Block 4
- **Inputs:** Regime probabilities (confidence), blended fraction, P3-D05
- **Formula:**
  ```
  IF regime_uncertain:
      f = min(f_shrunk, mean / (mean² + var))
  ```
  Robust (mean-variance) Kelly replaces standard Kelly when regime probability spread is low
- **Output:** Potentially reduced fraction
- **Dataset Refs:** P3-D05 (for mean/var computation)
- **Constraints:** Only activates when regime classification confidence is low

### Layer 5: AIM / MoE Modifier

- **Purpose:** Apply the combined intelligence modifier from all active AIMs
- **Where:** Online Block 4, using output from Online Block 3 (MoE aggregation)
- **Inputs:** combined_modifier from PG-23 (Online Block 3), P3-D01, P3-D02
- **Formula:**
  ```
  combined_modifier = Σ(modifier_a × DMA_weight_a) / Σ(DMA_weight_a)
  f_aim = f × combined_modifier
  ```
  combined_modifier bounded to [0.5, 1.5]
- **Output:** AIM-adjusted fraction
- **Dataset Refs:** P3-D01, P3-D02
- **Constraints:** Modifier floor 0.5, ceiling 1.5. Neutral (1.0) when no active AIMs or error.

### Layer 6: Risk Goal Adjustment

- **Purpose:** Scale fraction based on TSM account classification and risk goal
- **Where:** Online Block 4
- **Inputs:** P3-D08 (TSM state, risk_goal), account classification
- **Formula:**
  ```
  f_risk = f_aim × risk_goal_multiplier
  ```
- **Risk goal multipliers:**

| Risk Goal | Multiplier | Context |
|-----------|-----------|---------|
| PASS_EVAL | 0.5 -- 0.85 | Evaluation accounts: conservative to protect against MDD breach |
| PRESERVE_CAPITAL | 0.5 | Protective mode: minimal exposure |
| GROW_CAPITAL | 1.0 | No additional reduction (funded accounts, growth phase) |

- **Output:** Risk-adjusted fraction
- **Dataset Refs:** P3-D08
- **Constraints:** Multiplier range [0.5, 1.0]

### Layer 7: Contract Conversion + Caps

- **Purpose:** Convert fractional Kelly to integer contract count, apply all hard caps
- **Where:** Online Block 4
- **Inputs:** f_risk, P3-D08 (TSM caps), P3-D16 (capital silo), asset specs
- **Formula:**
  ```
  k = min(⌊f_risk × capital / risk_per_contract⌋, TSM_caps)
  ```
  Where risk_per_contract = SL_distance × point_value. Fee adjustment: R_eff = R + φ (adds roundtrip cost to risk)
- **Output:** Final integer contract count (or 0 = skip trade)
- **Dataset Refs:** P3-D08, P3-D16, P3-D00 (asset specs)
- **Constraints:** TSM position limits, capital silo max positions (5), account-level caps

### Kelly Pipeline Summary

```
P3-D03 trade outcome
   |
   v
L1: Raw Kelly per [asset][regime][session]  (Offline B8)
   |
   v
L2: Regime blend: f = Σ P(r) × f*_r        (Online B4)
   |
   v
L3: Shrinkage: f *= max(0.3, 1 - est_var)   (Offline B8 computes, Online B4 applies)

   |

       capped by TSM limits

```

## §6 — Circuit Breaker 5 Layers

The circuit breaker is a post-sizing, pre-signal risk screen that protects Topstep accounts from intraday drawdown breaches. It adds constraints -- never loosens existing ones. Non-Topstep accounts bypass entirely.

[SOURCE: Nomaan_Edits_P3.md (Isaac spec, 2026-03-12) + Topstep_Optimisation_Functions.md (full pseudocode)]

**Integration point:** After Kelly (Online B4) + Trade Selection (Online B5), before Signal Output (Online B6):
```
B4 (Kelly) -> B5 (Trade Selection) -> B7B (Circuit Breaker) -> B6 (Signal Output)
```

**PG ID:** P3-PG-27B (circuit_breaker_screen_A) -- Online Block 7

**Inputs:** SOD-locked params from P3-D08[ac].topstep_state | P3-D23 (intraday state) | P3-D25 (CB params per basket) | incoming signal

**Outputs:** take: True/False + reason code. Suppressed signals logged to P3-D23.

### SOD Parameters (computed at 19:00 EST daily by Command B8)

```
A = current_balance
mdd_pct = mdd_fixed / A
f(A) = p × mdd_pct                           (risk per trade %)
N = floor(e / f(A))                           (max trades per day)
E = e × A                                    (daily exposure $)
L_halt = c × E = c × e × A                   (hard halt threshold $)
```

Default parameters: p=0.005, e=0.01, c=0.5. For 150K account: L_halt = $750, N = floor(0.01/0.00003) = 333.

### Layer 0: Simultaneous Position Limit (XFA/Scaling Only)

- **Condition:** current_open_micros + proposed_micros > scaling_tier_micros
- **Action:** BLOCKED with reason "SCALING_CAP_EXCEEDED"
- **Scope:** XFA (evaluation) accounts with scaling plan active only. Live/funded accounts skip this layer.
- **Purpose:** Limits concurrent exposure, NOT daily volume. When positions close, slots free up.
- **Dataset Refs:** P3-D08 (scaling_tier_micros), Online B7 position tracker
- **Recovery:** Automatic when positions close

### Layer 1: Hard Halt (Account Survival)

- **Condition:** |L_t| + ρ_j >= L_halt (where L_halt = c × e × A)
- **Action:** BLOCKED with reason "HARD_HALT_PREEMPTIVE"
- **ρ_j:** Worst-case risk of proposed trade = contracts × (SL_distance × point_value + φ)
- **CRITICAL: PREEMPTIVE check** -- blocks trades whose worst-case SL outcome would breach the halt, not just trades where L_t has already breached. Prevents double-loss scenarios.
- **Example:** L_t = -$495, ρ_j = $495 (7 MES × $70.74). Check: |-495| + 495 = $990 >= $750. **BLOCKED.** Without preemptive check, trade would pass (|-495| < $750) and a second SL hit would push L_t to -$990.
- **Dataset Refs:** P3-D23 (L_t), P3-D08 (L_halt)
- **Parameters:** c = 0.5 (from P1 grid search or default)
- **Recovery:** Next trading day (SOD reset at 19:00 EST)
- **Active from day 1.** No data dependency.

### Layer 2: Remaining Budget

- **Condition:** n_t >= N (trades taken today >= max allowed)
- **Action:** BLOCKED with reason "BUDGET_EXHAUSTED"
- **Purpose:** Exposure cap. Remaining slots = N - n_t.
- **Dataset Refs:** P3-D23 (n_t), P3-D08 (N)
- **Parameters:** N = floor(e / f(A))
- **Recovery:** Next trading day (SOD reset)
- **Active from day 1.** No data dependency.

### Layer 3: Conditional Expectancy Filter (Per-Basket)

- **Condition:** μ_b <= 0 (expected return of next trade in this basket is non-positive)
- **Formula:** μ_b = r_bar_b + β_b × L_b
  - r_bar_b = unconditional mean return for basket b
  - β_b = loss-predictiveness coefficient (OLS from P3-D25)
  - L_b = cumulative P&L for basket b today (from P3-D23)
- **Action:** BLOCKED with reason "BASKET_NEGATIVE_EXPECTANCY"
- **Interpretation:** When β_b < 0, losing trades predict further losses. L* = -r_bar / β_b is the breakeven loss level. When |L_b| > L*, expected next return turns negative.
- **Dataset Refs:** P3-D25 (r_bar_b, β_b), P3-D23 (L_b)
- **Parameters:** β_b estimated by Offline B8 (PG-16C) via OLS regression on trade history
- **Recovery:** Next trading day (L_b resets)
- **Cold start:** β_b = 0 -> μ_b = r_bar_b > 0 always (assuming positive-expectancy strategy). **Layer disabled until 100+ observations.**

### Layer 4: Correlation-Adjusted Conditional Sharpe

- **Condition:** S <= λ (conditional Sharpe ratio below threshold)
- **Formula:**
  ```
  S = μ_b / (σ × sqrt(1 + 2 × n_t × ρ_bar))
  ```
  - σ = per-trade return std dev (from P3-D25)
  - n_t = trades taken today (from P3-D23)
  - ρ_bar = average within-day trade correlation (from P3-D25)
- **Action:** BLOCKED with reason "SHARPE_BELOW_THRESHOLD"
- **Interpretation:** As more trades are taken (n_t increases), portfolio variance grows (especially with correlated trades). The Sharpe denominator increases, making the threshold harder to meet. This naturally throttles trading as intraday correlation builds.
- **Dataset Refs:** P3-D25 (σ, ρ_bar), P3-D23 (n_t)
- **Parameters:** λ = Sharpe threshold (from P1 grid search or default 0)
- **Recovery:** Next trading day (n_t resets)
- **Cold start:** ρ_bar = 0 -> denominator = σ -> S = μ_b/σ ≈ unconditional Sharpe. **Layer disabled until 100+ observations** (with λ = 0, passes unless strategy is negative-expectancy).

### CB Parameter Estimation (Offline Block 8, PG-16C)

**P3-D25 schema per account per model:**
- account_id, model_m, r_bar, beta_b, sigma, rho_bar, n_observations, p_value, last_updated

**Estimation:** OLS regression r_i = r_bar + β_b × L_b_at_trade_time + ε
- Per basket per account
- Same trigger as Kelly updates: after each trade outcome batch, or daily minimum
- Only for accounts with topstep_optimisation: true

### Cold Start Behaviour Summary

| Layer | Cold Start (< 100 obs) | Active From |
|-------|----------------------|-------------|
| Layer 0 (Scaling) | Active if XFA | Day 1 |
| Layer 1 (Hard halt) | Active. c from P1 or default 0.5 | Day 1 |
| Layer 2 (Budget) | Active. N from SOD params | Day 1 |
| Layer 3 (Expectancy) | Disabled. β_b = 0 -> μ_b always positive | After 100+ obs |
| Layer 4 (Sharpe) | Disabled. ρ_bar = 0, λ = 0 -> always passes | After 100+ obs |

## Block 8: Daily Reconciliation

- **PG ID:** P3-PG-39 (`daily_reconciliation_A`)
- **Purpose:** Syncs broker balance as source of truth, computes SOD parameters, resets intraday state, evaluates XFA scaling, recommends payouts
- **Trigger:** 19:00 EST daily (SOD boundary)
- **Inputs:** Broker API (balance query) | P3-D08 (TSM) | P3-D23 (intraday state) | P3-D16 (capital silos)
- **Outputs:** P3-D08 (updated balances + SOD params) | P3-D23 (reset) | P3-D19 (reconciliation log) | payout notifications
- **Shared vs Per-User:** Per-user, per-account

### Reconciliation Steps (per user, per account)

**Step 1 — Broker Balance Sync:**
- If API adapter active: query broker balance
- If `abs(broker_balance - system_balance) > reconciliation_threshold`: create incident `("RECONCILIATION", "P2_HIGH", "FINANCE", ...)`
- **Broker is source of truth:** `P3-D08[ac].current_balance = broker_balance`

**Step 2 — SOD Parameter Recalculation:**

```
A = P3-D08[ac].current_balance
mdd_pct = 4500 / A
risk_per_trade_eff = (4500 * p + phi) / A
max_trades = floor(e * A / (4500 * p + phi))
exposure_budget = e * A
halt_threshold = c * e * A
```

**[V3 AMENDMENT]** (from `Nomaan_Edits_P3.md`, Change 1 — Isaac, 2026-03-12): SOD computation applies only to accounts with `topstep_optimisation: true` in TSM. Extended parameters include:

```
tsm.topstep_state = {
    mdd_pct:              mdd_fixed / A,
    risk_per_trade_pct:   p * (mdd_fixed / A),
    risk_per_trade_dollar: mdd_fixed * p,
    max_trades:           floor(e / (p * mdd_fixed / A)),
    daily_exposure:       e * A,
    hard_halt_threshold:  c * e * A,
    max_payout:           min(5000, 0.5 * max(A - 150000, 0)),
    post_payout_mdd_pct:  mdd_fixed / (A - max_payout) if A > 150000 else mdd_fixed / A
}
```

Where `p`, `e`, `c` come from `tsm.topstep_params` (P1 grid search or pseudotrader).

**Step 3 — Reset Intraday State:**
- `P3-D23[ac].L_t = 0` (cumulative P&L today)
- `P3-D23[ac].n_t = 0` (trades today)
- `P3-D23[ac].session_trades = []`

**Step 4 — XFA Scaling Tier Update:**
- If `scaling_plan_active`: compute profit = A - starting_balance
- `scaling_tier_micros = lookup_scaling_tier(profit)`

**Step 5 — Payout Recommendation:**
- Compute `payout_decision(A, profit, tsm_configs[ac].payout_rules)`
- If withdraw recommended: notify user at priority MEDIUM

**Step 6 — Log Reconciliation:**
- Append to `P3-D19`: user, accounts, timestamp, mismatches, corrections

### D00: asset_universe_register

- **Purpose:** Master registry of all assets in the trading universe with status, strategy references, data sources, and instrument specifications.
- **Key Fields:** `asset_id` (string), `p1_status` (string: VALIDATED/PENDING/NOT_RUN), `p2_status` (string), `captain_status` (string: WARM_UP/ACTIVE/DECAYED/DATA_HOLD/ROLL_PENDING/PAUSED), `warm_up_progress` (float 0.0–1.0), `aim_warmup_progress` (dict: {aim_id: float}), `locked_strategy` (reference → P2-D06), `roll_calendar` (object: {current_contract, next_contract, next_roll_date, roll_confirmed}), `exchange_timezone` (string IANA), `point_value` (float), `tick_size` (float), `margin_per_contract` (float), `session_hours` (object: {NY/LON/APAC → {open, close}}), `p1_data_path` (string), `p2_data_path` (string), `data_sources` (object — see sub-schema), `data_quality_flag` (string: CLEAN/PRICE_SUSPECT/ZERO_VOLUME/VOLUME_EXTREME/STALE_FEATURE), `created` (datetime), `last_updated` (datetime)
- **Sub-schema — data_sources:** Per-asset nested object mapping each data feed to an adapter config: `price_feed`, `options_chain`, `vix_feed`, `cot_data`, `economic_calendar`, `macro_data`, `cross_asset_prices`. Each entry has: `adapter` (REST/WEBSOCKET/FILE/BROKER_API), `endpoint` (string), `frequency` (STREAMING/POLL_10S/POLL_1MIN/DAILY_FILE/DAILY_PRE_SESSION/WEEKLY), `auth_ref` (vault reference), `provides` (list of data keys).
- **Writer:** Command (onboarding); Offline (asset_bootstrap, status transitions)
- **Reader:** Online B1 (session data ingestion, health check); all processes (asset lookup)
- **Write Frequency:** On-event (asset onboarding, status change, roll detection)
- **Retention:** Permanent (no TTL)
- **Notes:** Adding a new asset requires ALL `data_sources` entries populated. Command B10 validates config before acceptance. Null adapter = AIM outputs neutral 1.0 for that asset. Online B1 validates each source at session open; failure → DATA_HOLD flag.

### D01: aim_model_states

- **Purpose:** Persisted state for each AIM module (1–15) including trained model, warm-up progress, and current modifier values.
- **Key Fields:** `aim_id` (int 1–15), `status` (string: INSTALLED/COLLECTING/WARM_UP/BOOTSTRAPPED/ELIGIBLE/ACTIVE/SUPPRESSED), `model_object` (serialised), `warmup_progress` (float 0.0–1.0), `current_modifier` (dict: {asset_id: float}), `last_retrained` (datetime), `missing_data_rate_30d` (float)
- **Writer:** Offline B1 (AIM training and management)
- **Reader:** Online B3 (AIM aggregation — reads current_modifier per asset)
- **Write Frequency:** Per-retrain event (varies by AIM: daily to monthly)
- **Retention:** Latest state per AIM; version history in D18
- **Notes:** AIM-16 (HMM) state is stored separately in D26, not here. `model_object` is AIM-specific serialised state vector. Obsidian schema has additional fields: `asset_id` (SYMBOL, nullable for session-level AIM-16), `state_json` (STRING), `version` (STRING).

### D02: aim_meta_weights

- **Writer:** Offline B1.4 (DMA meta-learning update)

- **Retention:** Latest state; version history in D18

### D03: trade_outcome_log

- **Purpose:** Complete record of every resolved trade including P&L, fees, slippage, regime context, and AIM state at entry. Primary feedback signal for all learning loops.
- **Key Fields:** `trade_id` (UUID), `user_id` (string), `asset` (string), `direction` (int: +1 LONG / -1 SHORT), `entry_price` (float), `signal_entry_price` (float), `exit_price` (float), `contracts` (int), `gross_pnl` (float), `commission` (float), `pnl` (float — net, feeds learning), `slippage` (float — computed: (actual_entry - signal_entry) × direction × contracts × point_value), `outcome` (string: TP_HIT/SL_HIT/MANUAL_CLOSE/TIME_EXIT), `entry_time` (datetime), `exit_time` (datetime), `regime_at_entry` (string: LOW_VOL/HIGH_VOL), `aim_modifier_at_entry` (float), `aim_breakdown_at_entry` (dict: {aim_id: {modifier, weight, reason_tag}}), `session` (int: 1=NY, 2=LON, 3=APAC), `account` (string), `tsm_used` (string)
- **Writer:** Online B7 (position monitoring → resolve_position)
- **Reader:** Offline orchestrator (triggers B1 DMA, B2 BOCPD, B8 Kelly/EWMA updates); Command B8 (reconciliation); GUI (trade history)
- **Write Frequency:** Per-trade resolution
- **Retention:** Permanent (audit trail)
- **Notes:** Published to Redis `captain:trade_outcomes` channel. Offline subscribes. This is the single most important feedback dataset — all learning loops (DMA, BOCPD, EWMA, Kelly) consume it. Obsidian schema uses simpler types: `strategy_type` (SYMBOL), `session_id` (STRING).

- **Notes:** Obsidian schema uses lighter types: `detector_id` (SYMBOL: L2/L3), `hazard` (DOUBLE), `level` (INT). V3 provides the full nested structure.

### D05: ewma_states

- **Purpose:** EWMA-smoothed win rate, average win, and average loss per asset × regime × session. Direct inputs to Kelly fraction computation.
- **Key Fields:** `[asset][regime][session].win_rate` (float — EWMA of win indicator), `[asset][regime][session].avg_win` (float — EWMA of $/contract on wins), `[asset][regime][session].avg_loss` (float — EWMA of $/contract on losses, positive value)
- **Indexing:** asset_id × regime (LOW_VOL, HIGH_VOL) × session (1=NY, 2=LON, 3=APAC). All values normalised to per-contract dollar terms.
- **Writer:** Offline B8 (Kelly parameter updates — EWMA update step)
- **Reader:** Online B4 (Kelly Layer 1 — regime-conditional raw Kelly uses win_rate, avg_win, avg_loss)
- **Write Frequency:** Per-trade outcome
- **Retention:** Latest state; version history in D18
- **Notes:** Obsidian schema adds `ewma_mean` (DOUBLE), `ewma_var` (DOUBLE), `lambda` (DOUBLE — decay parameter). Category A learning — updates from ALL signals across instances. Bootstrapped from P1 trade history (60 rows for 10 assets × 2 regimes × 3 sessions).

## §10 — Supporting Systems

### Fee Resolution [SOURCE: Obsidian doc 25]

**Purpose:** TSM fee schedules, commission resolution, TopstepX sizing optimisation, payout rules (XFA vs Live), and funded account constraints.

#### Fee Schedule Structure

Each TSM profile carries a `fee_schedule` JSON:

| Field | Type | Meaning |
|-------|------|---------|
| `asset_pattern` | string/regex | Match symbol |
| `commission_per_contract` | double | Per fill side |
| `exchange_fees` | double | Pass-through |
| `slippage_ticks` | double | Expected slip |
| `currency` | string | USD |

**Supported profiles:** TOPSTEP_EXPRESS (futures per-side), TOPSTEP_LIVE (same structure, may differ rates), IBKR (per-exchange schedule).

**Functions:**
- `resolve_commission(tsm, asset, contracts)` → total $ fees for trade (both sides if round-turn)
- `get_expected_fee(tsm, asset)` → unit expected fee for pre-trade Kelly / CB checks

**Fallback:** If no `fee_schedule` on TSM → `commission_per_contract × contracts × 2` (explicit round-turn assumption).

#### TopstepX Sizing Optimisation (XFA Framing)

| Symbol | Meaning |
|--------|---------|
| **A** | Account balance / eval size anchor |
| **p** | Per-contract risk parameter |
| **φ** | Per-round fixed cost (fees + slip in $) |
| **e** | Max equity loss fraction before halt |
| **c** | DLL cushion multiple |

**Formulas:**
- Contracts function: `f(A) = 4500 / A` (illustrative constant per programme)
- Effective risk: `R_eff = (4500·p + φ) / A`
- Max contracts: `N = floor(e·A / (4500·p + φ))`
- Max loss budget: `E = e·A`
- Halt line: `L_halt = c·e·A`

**SOD boundary:** 19:00 EST — daily resets / metrics. Compounding: profitable day increases A → ALL derived params recompute.

**Integration:** `final_contracts = min(kelly_contracts, tsm_cap, topstep_daily_cap, scaling_cap)` where `scaling_cap = tier_micros − current_open_micros`.

#### XFA $150K Scaling Tiers

Evaluated end-of-day, applies next session. **XFA only** — Live accounts do NOT use scaling.

| Profit Level | Max Minis | Max Micros |
|-------------|-----------|------------|
| < $1,500 | 3 | 30 |
| $1,500 – $2,000 | 4 | 40 |
| $2,000 – $3,000 | 5 | 50 |
| $3,000 – $4,500 | 10 | 100 |
| > $4,500 | 15 | 150 |

**10:1 ratio:** 1 mini = 10 micros. Any combination totalling ≤ tier limit in mini-equivalents.

#### Payout Rules

**XFA payout:**
- Cap: max $5,000 OR 50% of profit (whichever policy wins per programme)
- Commission: 10% on payout
- Winning days required: 5
- Tier floor: $4,500 minimum tier balance

**Live funded payout:**
- Winning days: 30 → daily payout eligibility
- Commission: 0%
- Withdrawal: up to 100% of balance per provider policy
- GUI notification on eligibility change

**Function:** `payout_decision(A, profit, W_max, tier_floor, commission_rate)` → `{approved_amount, withholding, reason_code}`

#### Live Funded Account Rules

|------|--------|

| DLL | $4,500; $2,000 if tradable balance ≤ $10k |

### Signal Distribution [SOURCE: Obsidian doc 20]

**Purpose:** Anti-copy-trading distribution for prop accounts. Ensures two users NEVER hold the same `(asset, direction)` in the same session window.

#### Placement

Between Block 5B (quality gate) and Block 6 (signal output) — new PG-30.
- **V1:** No-op (single-user passthrough)
- **V2+:** Auto-activates when `len(active_users) > 1`

#### Scope

| User Type | Distribution |
|-----------|-------------|
| Prop accounts only | Full rules apply |
| Broker-only users | Bypass distribution |
| Mixed (prop + broker) | Participate for ALL accounts |

#### Conflict Key

`(asset, direction, session_window)` — at most one assignee per conflict key.

#### Algorithm (7 Steps)

1. Merge per-user recommended signals
2. Classify prop-tradeable vs broker-only
3. Sort by `expected_edge`
4. Priority rotation — circular queue (P3-D27)
5. Assign each conflict key to exactly one user
6. EV balancing — cumulative EV tracking across sessions (rolling 30d)
7. Append broker-only signals to ALL qualifying users after distribution

#### Invariant

Two users must **NEVER** hold the same `(asset, direction)` in the same session window. This is the hard constraint that drives the entire distribution system.

#### Auto-Updating

No internal parameters require manual tuning. System fully adapts to upstream changes: AIM weights, Kelly parameters, BOCPD, strategy injection, user join/leave, TSM changes, HMM regime shifts.

#### What Does NOT Change

Blocks 1–3 (data/regime/AIM), Blocks 4–5B (per-user pipeline), AIM framework, CB layers, Kelly formula, signal output format (Block 6), monitoring, decay detection, and pseudotrader — all unchanged by signal distribution.

### Pseudotrader System [SOURCE: Obsidian doc 28]

**Purpose:** Account-aware pseudotrading for PG-09/09B/09C — replay historical (or synthetic) trades under per-account constraints to evaluate proposed updates before they affect live/eval accounts.

#### Problem Statement

Legacy pseudotrader ignores account-specific rules (DLL, scaling tiers, Live vs XFA differences). A strategy can show paper improvement while violating real max drawdown, daily loss, contract caps, or session rules. **Requirement:** Replays must apply the same constraint bundle as the target account class.

#### account_config Input (from P3-D08)

| Field | Role |
|-------|------|
| `max_drawdown_limit` | Trailing MDD cap; null for Live (no trailing MDD) |
| `max_daily_loss` | DLL ceiling |
| `max_contracts` | Hard position size cap (when not on XFA scaling) |
| `scaling_plan` | XFA only: tiered contract limits vs balance/eval stage |
| `consistency_rule` | XFA: consistency gate (profit concentration) |
| `starting_balance` | Live: starting_tradable = 20% of transferred capital |
| `capital_unlock` | Live: each $9k profit unlocks next 20% of capital |
| `trading_hours` | session_open 18:00 EST, flat_by 16:10 EST, eod_exit_buffer 15:55 EST |

#### Replay Loop (Per Day, 11 Steps)

1. **SOD reset** — reset daily counters/flags for account type
2. **Load state** — balance, peak balance, daily PnL, scaling tier, unlock state
3. **Contract limit** — XFA: from `scaling_plan`; else: `max_contracts`
4. **Pre-trade MDD** — XFA/Eval: trailing MDD vs `max_drawdown_limit`; Live: tradable balance > 0
5. **DLL check** — block or resize if daily loss would breach `max_daily_loss`
6. **Trading hours** — no new risk outside `trading_hours`; enforce eod_exit_buffer / flat_by
7. **Size constraint** — clip to min(contract limit, risk-allowed, DLL headroom)
8. **Execute** — apply fill model with constrained size
9. **Post-trade** — update balance, max_balance, daily_pnl
10. **Consistency** — XFA: run consistency check post-trade
11. **Capital unlock** — Live: evaluate $9k milestones → advance 20% unlock tranche

#### Run Per Account Type

Separate replay per active account configuration (XFA, Eval, Live profile). Each produces: performance metrics, constraint breach event log, account-specific ADOPT or REJECT recommendation.

#### Eval Account Extensions

- Pass/fail: eval profit target tracked to completion or breach
- DLL breach treated as fail (eval terminated)
- Other eval rules: mapped into same replay scaffold as XFA where applicable

#### Deterministic Tick Stream

For synthetic regression tests when historical data unavailable:
- **Reproducibility:** SHA256-based mixer seeds all randomness → bitwise replay given same inputs
- **Sessions:** Calendar-based session generation (aligns with trading_hours)
- **Mid price:** Log-price geometric Brownian motion (GBM)
- **Microstructure:** Bid/ask spread model layered on mid

#### Replay Modes

| Mode | Behaviour |
|------|-----------|
| **LEGACY** | Single fixed MDD; no per-account constraints. Non-production label only. Backward comparison only. |
| **IDEAL** | Full account-aware replay: DLL, scaling, Live vs XFA/Eval, hours, consistency, capital unlock. **Default for go/no-go decisions.** |

#### Parity Interaction

Pseudotrader is listed in §9 D27 cross-reference as unchanged by signal distribution. It operates on historical/synthetic data and does not participate in live signal routing. However, Category A learning datasets (D02, D04, D05, D12) that feed into pseudotrader inputs ARE updated from all signals (including shadow-monitored theoretical outcomes from the other parity instance).

## §11 — Feedback Loops

Six closed-loop feedback circuits connect the three Captain processes. All loops originate from trade outcomes written to P3-D03 (trade_outcome_log) by Online B7 (Position Monitor).

**Sources:** Programs 1-2-3.canvas (feedback loops node), VALIDATE Programs 1-2-3.canvas (LOOP 1–6 section), Cross_Reference_PreDeploy_vs_V3.md (SOD parameters).

---

### Loop 1: AIM Meta-Learning

- **Trigger:** Trade close (Online B7 writes P3-D03)
- **Data Flow:**
  1. Online B7 PG-27 closes position → writes trade outcome to **P3-D03** (QuestDB)
  2. Trade outcome published on `captain:trade_outcomes` Redis channel
  3. Offline B1 PG-02 (DMA updater) reads P3-D03 → computes posterior inclusion probability: P(AIM|data) ∝ P^λ × L(x|AIM), λ=0.99
  4. Offline B1 PG-03 (HDWM diversity) enforces seed-type diversity — force-reactivates if all AIMs in a seed type suppressed
  5. Offline B1 PG-04 (ADWIN drift) detects distribution shift → weight *= 0.5 on drift
  6. Updated meta-weights written to **P3-D02** (QuestDB)
  7. Online B3 PG-23 reads P3-D02 → MoE aggregation: combined_mod = Σ(mod × DMA_weight) / Σ(weights), clamped [0.5, 1.5]
  8. Combined modifier flows to Online B4 (Kelly sizing) → next signal reflects updated AIM beliefs
- **Datasets R/W:** P3-D03 (read), P3-D02 (write/read), P3-D01 (write — lifecycle state), P3-D04 (write — drift flags)
- **Latency:** Async, per trade outcome. Offline processes event within seconds; Online reads updated weights at next session.
- **Cross-Process:** Online → Offline → Online
- **Category:** A (learns from ALL signals, including shadow/theoretical outcomes in multi-instance mode)

---

### Loop 2: Decay Detection → Strategy Injection

- **Trigger:** Trade close (Online B7 writes P3-D03)
- **Data Flow:**
  1. Online B7 PG-27 closes position → writes to **P3-D03**
  2. Offline B2 PG-05 (BOCPD) computes changepoint posterior: cp = posterior[0], NIG conjugate → Student-t predictive
  3. Offline B2 PG-06 (CUSUM) runs 2-sided: C⁺ = max(0, C⁺ + x - k)
  4. Offline B2 PG-07 runs bootstrap calibration
  5. Offline B2 PG-08 (decay response) applies tiered response:
     - **Level 2 (cp > 0.8):** Writes sizing_override to **P3-D12** → Online B4 Kelly reduced
     - **Level 3 (cp > 0.9 sustained 5d):** Writes DECAYED to **P3-D00** → triggers full retest chain:
       - Offline B3 (pseudotrader) retests strategy → writes **P3-D11**
       - Offline B4 (injection) compares: E_new > 1.2×E → Adopt; 0.9–1.2× → PARALLEL; else → Reject
       - If adopt: Offline B4 PG-11 transitions with weight ramp: w_new(d) = d/T → updates **P3-D00**, **P3-D06**
       - Notification sent to user via `captain:alerts`
  6. Updated decay state written to **P3-D04** (QuestDB)
- **Datasets R/W:** P3-D03 (read), P3-D04 (write), P3-D12 (write — L2), P3-D00 (write — L3), P3-D06 (write — injection history), P3-D11 (write — pseudotrader results)
- **Latency:** Per trade (L2 response immediate). L3 requires 5 consecutive days above threshold.
- **Cross-Process:** Online → Offline → Online (L2: sizing override), Offline → Offline → User → Offline (L3: injection chain)
- **Category:** A (decay detection uses ALL signals)

---

### Loop 3: Kelly EWMA Parameter Updates

- **Trigger:** Trade close (Online B7 writes P3-D03), processed per trade batch or daily
- **Data Flow:**
  1. Online B7 writes trade outcome to **P3-D03**
  2. Offline B8 PG-15 (Kelly EWMA) reads P3-D03 + reads **P3-D04** (changepoint probability for adaptive α)
  3. Computes adaptive EWMA: α = f(cp) — higher α (faster adaptation) when changepoint detected
  4. Updates per-cell estimates: p̂ (win probability), W̄ (average win), L̄ (average loss)
  5. Computes Kelly fraction: f* = (pW - (1-p)L) / W
  6. Writes updated EWMA states to **P3-D05** (QuestDB)
  7. Writes updated Kelly parameters to **P3-D12** (QuestDB)
  8. Online B4 PG-24 reads P3-D05 + P3-D12 at next session → 7-layer Kelly pipeline uses updated f*
- **Datasets R/W:** P3-D03 (read), P3-D04 (read — cp for adaptive α), P3-D05 (write), P3-D12 (write)
- **Latency:** Per trade batch or daily. Updates available for next session.
- **Cross-Process:** Online → Offline → Online
- **Category:** A (learns from ALL signals — pooled across multi-instance)

---

### Loop 4: β_b Learning (Circuit Breaker Layer 3)

- **Trigger:** Trade close (Online B7 writes P3-D03), per trade batch
- **Data Flow:**
  1. Online B7 writes trade outcome to **P3-D03**
  2. Offline B8 PG-16C (β_b estimator) runs OLS regression on per-basket loss serial correlation
  3. Estimates: μ_b = r̄ + β_b × L_b (expected return conditional on prior basket loss)
  4. Computes critical loss threshold: L* = −r̄ / β_b
  5. Writes β_b parameters to **P3-D25** (QuestDB)
  6. Online Circuit Breaker Layer 3 reads P3-D25 → IF μ_b ≤ 0: BLOCK signal with reason
- **Datasets R/W:** P3-D03 (read), P3-D25 (write/read)
- **Latency:** Per trade batch. Cold-start: β_b = 0 when n < 100 trades (Layer 3 effectively disabled).
- **Cross-Process:** Online → Offline → Online
- **Category:** B (learns from TAKEN trades only — per-instance independent in multi-instance mode)

---

### Loop 5: Intraday Circuit Breaker State

- **Trigger:** Trade close (Online B7), immediate within same session
- **Data Flow:**
  1. Online B7 PG-27 closes position → updates **P3-D23** (QuestDB): L_t += pnl, n_t++
  2. Online Circuit Breaker reads P3-D23 in real-time:
     - **L0 (Scaling Cap):** XFA only — micros > tier_limit? Reads P3-D08 tier limits.
     - **L1 (Preemptive Halt):** |L_t| + ρ_j ≥ L_halt, where ρ_j = c × (SL × pv + φ). Reads P3-D23 + fee_schedule.json.
     - **L2 (Budget Check):** E − |L_t| sufficient? Reads P3-D08 (E, L_halt) + P3-D23 (L_t).
  3. IF any layer breached → BLOCK signal with reason
  4. **RESET:** 19:00 ET daily — Command B8 PG-39 (reconciliation) zeros P3-D23
- **Datasets R/W:** P3-D23 (write/read — intraday state), P3-D08 (read — SOD parameters), fee_schedule.json (read)
- **Latency:** Immediate (same session, intra-trade). Fastest feedback loop.
- **Cross-Process:** Online → Online (intra-session). Command resets at SOD.
- **Category:** B (uses actual account P&L — per-instance)

### Loop 6: SOD Compounding (Daily Reconciliation)

- **Trigger:** Daily at 19:00 ET (Command B8 PG-39 cron)
- **Data Flow:**
  1. Command B8 PG-39 (reconciliation) reads broker API → gets current account balance A
  2. Computes TopstepX-optimised SOD parameters:
     - f(A) = 4500 / A (risk fraction)
     - R_eff = (4500p + φ) / A (effective risk per trade including fees)
     - N = ⌊eA / (4500p + φ)⌋ (max trades per day)
     - E = eA (daily exposure budget)
     - L_halt = ceA (halt threshold)
  3. Updates **P3-D08** (QuestDB) topstep_state with new SOD parameters
  4. Zeros **P3-D23** (QuestDB) — intraday CB state reset
  5. Generates payout recommendation (if applicable)
  6. A compounds daily — next day uses updated balance
  7. Online B4 PG-24 reads P3-D08 at next session → Kelly pipeline Layer 7 applies updated caps
- **Datasets R/W:** P3-D08 (write — topstep_state), P3-D23 (write — zeroed), broker API (read)
- **Latency:** Daily at 19:00 ET. Parameters effective for next trading day.
- **Cross-Process:** Command → Online
- **Category:** B (uses actual account balance — per-instance)

### P2 Strategy Selection Algorithm (how locked strategies were chosen)

P2 Block 3a PG-03 selects the top strategy per asset using a composite score:

```
CS = OO × ln(N) × (1 + |τ|)
```

Where:
- **OO** = P1 Block 5 output (D-24) — sigmoid of weighted IR composite ∈ (0, 1)
- **N** = number of OOS trades (favours strategies with more evidence)
- **|τ|** = regime sensitivity from P2-D03 (Kendall τ-b, regime-conditioned performance)

Top-1 strategy per asset is locked into P2-D06 → read-only by P3.
